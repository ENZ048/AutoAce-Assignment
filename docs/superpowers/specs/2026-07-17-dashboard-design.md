# AutoAce Trial ‚Äî Hosted Evaluation Dashboard: Design

Date: 2026-07-17 ¬∑ Branch: `dashboard` ¬∑ Status: approved by user (backend Part 1 + frontend Part 2), pending spec review

## 1. Mission and scope

Build the hosted evaluation dashboard required by trial brief ¬ß7 (`../voice_tone_background_noise_dashboard_trial.pdf`): AutoAce logs in with provided credentials, uploads a batch (ZIP or folder of audio + one CSV manifest), sees a validation report **before** processing, watches live progress that survives page reloads, reviews per-file results in the exact 9-field schema, sees per-file errors with reasons, and downloads results as CSV/JSON. Dashboard = 10% of trial score; batch handling contributes to another 10% (production practicality).

**In scope (this session):** the complete dashboard app ‚Äî API + web UI ‚Äî connected to the existing `autoace_audio` backend, running locally.

**Out of scope (deferred):**
- Deployment/hosting (tentative target: a fresh isolated box; separate session), HTTPS, domain, credential handoff to AutoAce.
- Parallel per-file workers inside a batch (v1 processes files sequentially via `run_batch`; the speed-up path is a backend request for a `workers` param via `docs/DASHBOARD-BACKEND-REQUESTS.md`, or a web-layer mapper ‚Äî decision deferred).
- Audio playback in the review UI, cancel-mid-run, multi-user accounts, tone-arm selection in the UI (pipeline default is used).

**Hard constraints inherited:**
- Never modify `src/autoace_audio/`.
- Fully self-contained project: no code, assets, or references imported from any other project.
- No new third-party services (analytics, CDNs, trackers, storage). All libraries are self-hosted npm/PyPI packages.
- No verbatim customer speech in committed files or logs. Audio, results, and DB live only under gitignored `data/`.
- Client batches ‚âà 100 calls, up to ~500 MB ZIP; pipeline peaks ~4 GB per worker process; models are process-wide singletons.

## 2. Architecture overview

```
webapp/ (React SPA, Vite build)  ‚îÄ‚îÄHTTP‚îÄ‚îÄ‚ñ∂  src/dashboard/ (FastAPI)
                                              ‚îÇ  auth: env admin + JWT
                                              ‚îÇ  SQLite job store (data/dashboard.db)
                                              ‚îÇ  job dirs (data/jobs/<id>/)
                                              ‚îî‚îÄ‚ñ∂ worker process ‚îÄ‚îÄ‚ñ∂ autoace_audio.batch.run_batch()
```

One FastAPI process serves both the API (under `/api`) and the built SPA (static files from `webapp/dist`). Batch processing runs in a separate spawned worker process, one job at a time globally. All state lives in SQLite + the filesystem, so the UI is stateless and reload-safe.

### Repo layout (new code only)

```

tests/web/       # pytest suite (fast, stub analyze_fn, joins `make test`)
```

- Python (`[project.optional-dependencies] web`): `fastapi`, `uvicorn`, `python-multipart`, `bcrypt`, `pyjwt`. Core pipeline deps unchanged; install via `pip install -e .[web,dev]`.
- Webapp: `react`, `react-dom`, `react-router-dom`, `axios`, `react-hot-toast`, `lucide-react`, `@fontsource/sora` (self-hosted brand font), Tailwind CSS, Vite. No other runtime deps; no CDN assets.

| Key | Meaning |
|---|---|
| `DASHBOARD_ADMIN_USER` | single admin username |
| `DASHBOARD_ADMIN_PASSWORD_HASH` | bcrypt hash (generated by `python -m dashboard.hash_password`); plaintext never stored |

## 3. Job model and lifecycle

### States

`validating ‚Üí awaiting_confirmation ‚Üí queued ‚Üí running ‚Üí completed | failed`
plus `interrupted` (server restarted mid-run; re-runnable). Terminal states: `completed`, `failed`, `interrupted`.

### Storage

- `data/jobs/<job_id>/upload/` ‚Äî raw uploaded ZIP (kept for provenance; absent for folder uploads)
- `data/jobs/<job_id>/extracted/` ‚Äî the batch dir handed to `validate_batch`/`run_batch`
- `data/jobs/<job_id>/out/` ‚Äî `results.csv`, `results.json`, `errors.csv` written by `run_batch`, served verbatim
- `data/dashboard.db` ‚Äî SQLite, WAL mode

```sql
CREATE TABLE jobs (
  id TEXT PRIMARY KEY,               -- uuid4 hex; UI shows first 8 chars
  created_at TEXT NOT NULL,          -- ISO 8601 UTC
  original_name TEXT NOT NULL,       -- zip filename, or "folder upload (N files)"
  status TEXT NOT NULL,
  total INTEGER NOT NULL DEFAULT 0,
  done INTEGER NOT NULL DEFAULT 0,
  current_file TEXT,
  warnings TEXT NOT NULL DEFAULT '[]',  -- JSON array of validate_batch + run_batch warnings
  error TEXT,                        -- terminal failure reason
  started_at TEXT,
  finished_at TEXT,
  results_count INTEGER,
  errors_count INTEGER,
  worker_pid INTEGER
);
```

Write discipline: the API process owns status transitions; the worker process writes only progress fields (`done`, `current_file`) and its final `completed/failed` transition. WAL mode handles the two writers.

### Flow

1. **Upload** ‚Äî `POST /api/jobs` (multipart). Either exactly one `.zip`, or N files from a folder selection (a single non-ZIP file ‚Üí 400). Streamed to disk in chunks ‚Äî never whole-in-RAM; cumulative byte count enforced against `DASHBOARD_MAX_UPLOAD_MB` ‚Üí 413 + cleanup on breach. Folder uploads stream straight into `extracted/` (no archive step; `upload/` is absent). ZIPs are extracted by `zipsafe.py`: every member's resolved path must stay inside `extracted/` (rejects absolute paths and `..`; entries written as regular files only). Batch-root resolution mirrors the backend's `_unzip_if_needed` semantics (if the ZIP root has only a subdir + CSVs, the subdir becomes the batch root and CSVs move into it) ‚Äî reimplemented only because `extractall` lacks path safety and we control the destination dir.
2. **Validate** ‚Äî `autoace_audio.batch.validate_batch(extracted_dir)` runs synchronously (fast, no models). File list + warnings stored on the job; status ‚Üí `awaiting_confirmation`. Nothing processes yet ‚Äî the brief requires the validation report before processing. A missing manifest is a warning (backend semantics: process every audio file), not an error.
3. **Confirm** ‚Äî `POST /api/jobs/{id}/start` ‚Üí `queued`. A dispatcher task (FastAPI lifespan) starts the oldest queued job **only when no job is running** ‚Äî one batch at a time globally, protecting an 8 GB box from concurrent 4 GB model peaks. Others wait in `queued`.
4. **Run** ‚Äî dedicated worker process (`multiprocessing`, spawn context) calls `run_batch(extracted_dir, out_dir, progress_cb=‚Ä¶)` with the pipeline's default tone arm. `progress_cb(done, total, name)` updates SQLite once per file (‚âà100 writes per batch). `run_batch` re-validates internally (cheap, identical result) and provides per-file failure isolation; its end-of-run warnings (e.g. tone-arm fallback count) are appended to the job's warnings. Worker writes `completed` + counts on success; uncaught worker death is detected by the dispatcher (process exit without terminal status) ‚Üí `failed` with exit reason.
5. **Review/download** ‚Äî results read from `out/` artifacts (single source of truth). Downloads are the backend-written files byte-for-byte.
6. **Restart safety** ‚Äî on startup, any `running`/`queued` job whose `worker_pid` is dead ‚Üí `interrupted`; a job stuck in `validating` (server died mid-extract) ‚Üí `failed` with that reason. UI offers **Re-run** (`POST /api/jobs/{id}/rerun` ‚Üí re-enqueues the same `extracted/`; out/ is overwritten; Gemini re-spend for a full batch ‚âà $0.30 ‚Äî acceptable, and re-running is explicit user action).

## 4. API surface

All routes under `/api`; every route except `POST /api/auth/login` requires `Authorization: Bearer <JWT>`.

| Route | Purpose |
|---|---|
| `POST /api/auth/login` | `{username, password}` ‚Üí `{access_token}`; bcrypt verify; ~0.5 s delay on failure (basic brute-force friction) |
| `POST /api/jobs` | multipart upload (zip or files) ‚Üí job JSON in `awaiting_confirmation` with validation report |
| `GET /api/jobs` | job list (newest first) |
| `GET /api/jobs/{id}` | full job status ‚Äî the polling endpoint |
| `POST /api/jobs/{id}/start` | confirm validated batch ‚Üí `queued` |
| `POST /api/jobs/{id}/rerun` | re-enqueue an `interrupted`/`failed` job |
| `DELETE /api/jobs/{id}` | remove job row + directory; 409 while `queued`/`running` |
| `GET /api/jobs/{id}/results` | rows parsed from `out/results.json` for the table |
| `GET /api/jobs/{id}/errors` | rows parsed from `out/errors.csv` |
| `GET /api/jobs/{id}/download/{artifact}` | `artifact ‚àà {results.csv, results.json, errors.csv}`; FileResponse of the on-disk artifact |

Downloads are fetched by the SPA with the auth header and saved via blob URL ‚Äî tokens never appear in URLs or server logs.

## 5. Auth

Single admin user from env. Login verifies bcrypt hash, issues HS256 JWT (`sub`, `exp` = 24 h). A FastAPI dependency validates the token on every protected route. Stateless logout (client drops the token) ‚Äî documented limitation, appropriate for a single-user trial tool; secret + credential rotation happens at deployment/handoff (out of scope here).

## 6. Frontend

Fresh Vite + React 18 SPA in `webapp/`. Tailwind for styling, react-router for pages, axios instance with a request interceptor (Bearer from `localStorage`) and a 401 response interceptor (clear token ‚Üí redirect `/login`), react-hot-toast for feedback, lucide-react icons. Dev: Vite proxies `/api` ‚Üí `127.0.0.1:8000` (same origin, no CORS). Connected build: FastAPI serves `webapp/dist`.

### Screens

1. **Login** (`/login`) ‚Äî centered card, username + password, error toast, redirect to `/`.
2. **Jobs list** (`/`) ‚Äî **Upload batch** button; table: short id, name, created (local time), status chip, progress, actions (view/delete). Polls every 3 s while any job is non-terminal. Empty state documents the expected batch shape (audio files + one `name,result_json` CSV at root).
3. **Upload** ‚Äî drag-and-drop or picker; accepts one ZIP or a folder selection (files posted together); axios `onUploadProgress` bar; on response, navigate to the job page (validation report).
4. **Job detail** (`/jobs/:id`) ‚Äî renders purely from server state (reload-safe):
   - `awaiting_confirmation`: summary (N audio files ready; manifest found/absent), warnings listed verbatim and color-coded, **Start processing** / **Discard**.
   - `queued`/`running`: progress bar `done/total`, current filename, elapsed; 2 s poll.
   - `completed`: chips (succeeded / failed / warnings); results table ‚Äî `name` + the 9 schema fields (`emotional_tone`, `emotional_intensity`, `background_noise_present`, `background_noise_type`, `background_noise_severity`, `audio_quality`, `speaker_overlap_present`, `long_silence_present`, `confidence`) with enums as badges, booleans as ‚úì/‚Äî, confidence to 2 decimals, sticky header, quick text filter; errors table (filename + reason) prominent when non-empty; three download buttons.
   - `failed`/`interrupted`: reason + **Re-run**.

### Visual design (brand-matched to autoace.ai)

Direction extracted from the client's own site on 2026-07-17 (rendered screenshot + computed styles), so the dashboard reads as part of AutoAce's product rather than a third-party tool:

- **Tokens** (Tailwind theme extension): page background soft blue-tinted wash `#EFF4FA`; surfaces white with `#E5E7EB` hairline borders, 8‚Äì14 px radii, soft shadows; ink `#0F172A` (headings) / `#475569` (body); primary buttons dark navy `#020817` with white text (their "Get in Touch" style); accent blue `#2563EB`/`#3B82F6`; status tints matching their chips ‚Äî success `#15803D` on `#DCFCE7`, warning `#A16207` on amber-100, info `#1D4ED8` on `#DBEAFE`, error red-700 on red-100.
- **Type**: Sora 600/700 for headings and large numerals (`@fontsource/sora`, self-hosted); system-ui stack for body; `ui-monospace` for filenames, enum values, and confidence numbers. Their headline device ‚Äî leading phrase in blue, remainder in slate ‚Äî used sparingly on page titles.
- **Signature element** (the one memorable thing; everything else stays quiet): the running-job view is a **live analysis queue**, echoing the live-call-queue visualization in AutoAce's own hero ‚Äî three big Sora stat cards (analyzed / remaining / elapsed) above a file queue in which the current file pulses and completed files tick off. The evaluator watches their own product's visual metaphor analyzing their own calls.
- **Quality floor**: responsive down to tablet widths, visible keyboard focus rings, `prefers-reduced-motion` collapses the pulse to a static highlight, WCAG AA contrast on all text and badges.

## 7. Error handling and security

- Upload too large ‚Üí 413 with the configured limit in the message; partial files cleaned up.
- Hostile ZIP (path traversal, absolute paths) ‚Üí 400, nothing extracted.
- Wrong upload shape (single file that isn't a ZIP, or an empty upload) ‚Üí 400 naming the accepted inputs. Per-file content problems are validation's and `run_batch`'s job ‚Äî unmatched/undecodable files land in warnings or `errors.csv` and the UI errors table, never killing the batch.
- Worker crash ‚Üí `failed` with exit description; server restart ‚Üí `interrupted` + re-run.
- Disk hygiene: job artifacts persist for the trial (‚âà500 MB/batch; manual cleanup documented); `DELETE` removes a job's directory.
- Logs carry filenames, job ids, statuses ‚Äî never audio content, transcripts, or credentials.
- SPA and API are same-origin; no cookies (Bearer only), so no CSRF surface; standard security headers on responses.

## 8. Testing

- **pytest (`tests/web/`, fast, joins `make test`)** ‚Äî stub `analyze_fn` injected through the runner (no models/network/keys):
  auth (ok / wrong password / missing / expired token); upload (fixture ZIP happy path; 413 cap; zip-slip fixture rejected; unsupported type); validation report surfacing (manifest row without file ‚Üí its exact warning string on the job); lifecycle (confirm ‚Üí running ‚Üí completed; progress rows advance; results/errors endpoints match fixtures; artifacts downloadable byte-equal); per-file failure isolation surfaced (stub raises for one file ‚Üí errors table row, batch completes); one-at-a-time dispatch (second job stays `queued`); restart orphan ‚Üí `interrupted` ‚Üí re-run works; delete blocked while running.
- **Frontend** ‚Äî components stay logic-thin; the status‚Üíview mapping and any pure helpers get vitest unit tests.
- **E2E smoke** ‚Äî one Playwright script: login ‚Üí upload tiny fixture ZIP ‚Üí confirm ‚Üí progress ‚Üí results table ‚Üí download, against a server with `DASHBOARD_STUB_ANALYZE=1`.
- **Final acceptance (manual, real)** ‚Äî real server, real pipeline, the 3 provided sample calls uploaded through the browser as a batch; results cross-checked against their known outputs; downloads verified.

## 10. Decisions and rationale

| Decision | Why |
|---|---|
| React SPA + FastAPI API (user-selected) over server-rendered/Streamlit | richer review UI; user preference; costs a build step, mitigated by single-origin static serving |
| Polling (2‚Äì3 s) over SSE/WebSockets | 1 user, ‚â§ 100 files, minute-scale jobs; simplest reload-safe mechanism |
| SQLite + spawned worker process over Redis/RQ/Celery | zero new infra; one job at a time is a feature (memory safety), not a limitation |
| Sequential files in v1 (35‚Äì70 min / 100 calls) | `run_batch` is inherently single-process; parallelism deferred (backend request or web-layer mapper) |
| bcrypt hash in env, single admin | no user DB needed for a one-credential trial; plaintext never stored |
| PyJWT over python-jose | lighter, maintained, sufficient for HS256 |
| Stub-analyze env flag | UI/E2E testing without models or spend; default off, clearly named |
| Brand-matched UI (user request) | palette/type/status-chip system lifted from autoace.ai's rendered styles; font self-hosted to honor the no-third-party-services rule |
| Extraction reimplemented in `zipsafe.py` | backend's `extractall` lacks zip-slip protection and targets tempdirs; root-resolution semantics mirrored exactly |
